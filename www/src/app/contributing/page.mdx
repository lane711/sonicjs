### Why This Approach?

Open source maintainers have limited time. By focusing on code contributions first:

- **You demonstrate real interest** - Actions speak louder than words
- **You learn the codebase** - Hands-on experience is the best teacher
- **We can evaluate fit** - Your code shows us your skills and style
- **It respects everyone's time** - Meetings are more productive after you're familiar with the project

### Step 2: Claim the Issue

Before starting work:

1. **Comment on the issue** to indicate you're working on it
2. **Wait for confirmation** from a maintainer (especially for larger features)
3. **Ask questions** if anything is unclear about the requirements

### Step 3: Submit Your Pull Request

Once your work is ready:

1. Ensure all tests pass
2. Follow our code standards (see below)
3. Write a clear PR description explaining your changes
4. Reference the issue number in your PR
5. Be responsive to feedback

## Good First Contributions

Not sure where to start? Here are some ideas:

### Low-Hanging Fruit

- **Fix typos** in documentation
- **Improve error messages** to be more helpful
- **Write tests** for existing untested code
- **Update dependencies** (with proper testing)

### Medium Complexity

- **Fix bugs** from the issue tracker
- **Improve documentation** with examples
- **Add configuration options** to existing features
- **Performance improvements** with benchmarks

### Higher Complexity

- **New features** (please discuss first in an issue)
- **Major refactoring** (requires maintainer approval)
- **New plugins** (see our plugin development guide)

## Working with Maintainers

### Communication Guidelines

**Do:**
- Be patient - maintainers are often volunteers
- Provide context and details in issues/PRs
- Ask questions publicly so others can benefit
- Accept feedback gracefully and iterate

**Don't:**
- Request meetings before contributing
- Expect immediate responses
- Take feedback personally
- Submit large changes without discussion first

### Response Times

- **Issue responses**: Usually within 24 hours during M-F and non-US holidays
- **PR reviews**: Within a few days at most
- **Meeting requests**: Only after meaningful contribution

### When to Request Deeper Involvement

After you've:
- Had at least one meaningful PR merged
- Shown consistent quality in your contributions
- Demonstrated understanding of the project's direction
- Been active in discussions and helpful to others

You can then reach out about:
- Regular contributor status
- Joining the core team
- Taking ownership of specific areas
- Other collaboration opportunities

## Development Setup

```bash {{ title: 'Clone & Install' }}
# Fork the repository on GitHub first

git clone https://github.com/YOUR_USERNAME/sonicjs.git

npm install
```

```bash {{ title: 'Development' }}
# Start development server
npm run dev

# Run tests
npm test

# Run tests in watch mode
npm run test:watch

# Lint code
npm run lint

# Format code
npm run format
```

```bash {{ title: 'Submit PR' }}
# Create a feature branch
git checkout -b feature/your-feature-name

# Make your changes, then commit
git add .
git commit -m "feat: add your feature description"

# Push to your fork
git push origin feature/your-feature-name

# Create PR on GitHub
```

## Code Standards

### General Guidelines

- **TypeScript**: All code must be in TypeScript with proper types
- **Testing**: New features must include tests
- **Documentation**: Public APIs must be documented
- **Formatting**: Use Prettier (runs automatically on commit)
- **Linting**: ESLint rules must pass

### Commit Messages

We use AI to generate commit messages - make them as detailed as possible. Reference issue numbers when applicable.

### Pull Request Checklist

Before submitting a PR, ensure:

- [ ] All tests pass (`npm test`)
- [ ] Code is formatted (`npm run format`)
- [ ] Linting passes (`npm run lint`)
- [ ] Changes are documented if needed
- [ ] PR description explains the changes
- [ ] Related issue is referenced
